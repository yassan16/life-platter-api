## 1. プロジェクト概要

### 一文説明

life_platter アプリのユーザーが安全にアカウントを作成・ログインし、自分のデータに継続的にアクセスできるようにする認証・認可機能。

### 背景・課題

- アプリは個人の料理記録という**プライベートなデータ**を扱うため、ユーザーの識別と認証が不可欠
- モバイルアプリからの利用を想定し、**毎回ログインを強いないセッション維持**が求められる
- セキュリティを確保しつつ、**ユーザーが意識せずに使える**シームレスな認証体験が必要

### 対象ユーザー

- 自炊を記録したい個人ユーザー（スマートフォン利用を主体とする）

---

## 2. ゴール / 成功指標

| # | ゴール | 成功指標 |
|---|--------|---------|
| G-01 | 誰でも簡単にアカウントを作れる | メールアドレスとパスワードだけで登録完了できる |
| G-02 | 一度ログインすれば長期間使い続けられる | トークン期限切れを意識せずアプリを使える |
| G-03 | 不正アクセスからユーザーデータを守る | 他ユーザーのデータに一切アクセスできない状態を維持する |

---

## 3. ユーザーストーリー

| ID | Who（誰が） | What（何をしたいか） | Why（なぜ） | 優先度 |
|----|------------|---------------------|------------|--------|
| US-01 | 新規ユーザーが | アカウントを登録したい | アプリを使い始めるため | Must |
| US-02 | 登録済みユーザーが | メール/パスワードでログインしたい | 自分のデータにアクセスするため | Must |
| US-03 | ログイン中ユーザーが | セッションを自動維持したい | 毎回再ログインを避けるため | Must |
| US-04 | ログイン中ユーザーが | ログアウトしたい | セッションを安全に終了するため | Must |
| US-05 | ログイン中ユーザーが | 自分のアカウント情報を確認したい | 表示名・メールアドレスを確認するため | Should |

## 4. 機能要件

### FR-01: ユーザー登録

- メールアドレスとパスワードでアカウントを新規作成できる
- 既に使用済みのメールアドレスでは登録できない（エラーを返す）
- パスワードは安全に保存する（技術詳細: [design.md - セキュリティ制約](design.md#制約事項)）

### FR-02: ログイン

- 登録済みのメールアドレスとパスワードでログインできる
- 誤った認証情報ではログインできない
- 連続した失敗リクエストを制限する（技術詳細: [design.md - 設定値](design.md#設定値)）

### FR-03: セッション維持（トークン更新）

- ログイン後、一定期間は再ログインなしで操作を継続できる
- セッションの仕組みの詳細: [token-guide.md](token-guide.md)

### FR-04: ログアウト

- ログアウト操作により、現在のセッションを即座に無効化できる
- ログアウト後は同一セッションでの操作ができなくなる

### FR-05: アカウント情報確認

- ログイン中のユーザーが自分のユーザーID・ユーザー名・メールアドレスを確認できる

## 5. 非機能要件

### セキュリティ

- パスワードは平文で保存しない
- セッション情報はサーバー側で無効化できる（強制ログアウト可能）
- ブルートフォース攻撃を防ぐためレート制限を設ける

### パフォーマンス

- ログイン・トークン更新は通常のAPI応答速度内で完了する

### 可用性

- 認証機能の障害はアプリ全体の利用不能に直結するため、他機能と同等以上の安定性を確保する

---

## 6. スコープ外

以下は現時点のスコープに含めない：

- メールアドレス確認（Email Verification）
- パスワードリセット機能
- ソーシャルログイン（Google / Apple / LINE など）
- 多要素認証（MFA）
- 管理者によるユーザー操作（停止・削除など）

---

## 7. 用語集

| 用語 | 説明 |
|------|------|
| アクセストークン | APIリクエストの認証に使用する短命トークン |
| リフレッシュトークン | アクセストークンを再発行するための長命トークン |
| トークンローテーション | リフレッシュトークン使用時に新しいリフレッシュトークンも発行し、古いものを無効化する仕組み |
| レート制限 | 一定時間内のリクエスト回数を制限し、ブルートフォース攻撃等を防ぐ仕組み |
