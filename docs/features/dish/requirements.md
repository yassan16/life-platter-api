## 1. プロジェクト概要

### 一文説明

ユーザーが日々作った料理を記録・管理し、いつ何を作ったか振り返ることができる料理記録・管理機能。

### 背景・課題

- 自炊を続けるモチベーション維持のために、**過去の料理を振り返る手段**が必要
- 料理の名前・日付・写真を一元管理できないと、記録が散逸してしまう
- 大量の記録が蓄積された際も**快適に閲覧・検索できる**性能が求められる

### 対象ユーザー

- 自炊を習慣化したい個人ユーザー（スマートフォンで日常的に記録する）

---

## 2. ゴール / 成功指標

| # | ゴール | 成功指標 |
|---|--------|---------|
| G-01 | 作った料理を手軽に記録できる | 名前・日付・写真の登録が短時間で完了できる |
| G-02 | 過去の料理を素早く振り返れる | カテゴリや日付で絞り込んで目的の料理を見つけられる / 記録件数が増えても一覧の読み込みが快適なまま維持される |
| G-03 | 記録の誤りを後から修正できる | 登録後でも名前・日付・カテゴリ・写真を更新できる |

| ID | Who（誰が） | What（何をしたいか） | Why（なぜ） | 優先度 |
|----|------------|---------------------|------------|--------|

| US-02 | ログイン済みユーザーが | 料理一覧を見たい | 過去の料理を振り返るため | Must |
| US-03 | ログイン済みユーザーが | 料理の詳細を確認したい | 特定料理の画像・登録日を確認するため | Must |

## 4. 機能要件

### FR-01: 料理の登録

- 料理名・作った日付を入力して料理を登録できる
- カテゴリを任意で設定できる
- 写真を最大3枚まで添付できる
- 画像アップロードの仕組み詳細: [s3-image-upload.md](s3-image-upload.md)

**正常系**: 201 Created — 料理が登録され、登録内容が返却される
**エラーケース**:
- 画像が3枚を超過している（400 IMAGE_LIMIT_EXCEEDED）
- display_order が不正（400 INVALID_DISPLAY_ORDER）
- 指定されたカテゴリが存在しない（422 CATEGORY_NOT_FOUND）
- S3オブジェクトが存在しない（422 S3_OBJECT_NOT_FOUND）

**関連ストーリー**: US-01, US-07

### FR-02: 料理一覧の取得

- 自分が登録した料理を一覧で確認できる
- 作った日付の新しい順に表示される
- 大量データでも性能が劣化しない（技術詳細: [design.md - カーソルベースページネーション](design.md#カーソルベースページネーション)）

**正常系**: 200 OK — ページネーション付きの料理一覧が返却される
**エラーケース**:
- カーソル値が不正（400 INVALID_CURSOR）

**関連ストーリー**: US-02

### FR-03: 料理詳細の確認

- 特定の料理の名前・日付・カテゴリ・添付写真をすべて確認できる
- 自分が登録した料理のみ閲覧可能

**正常系**: 200 OK — 指定した料理の詳細情報が返却される
**エラーケース**:
- 料理が存在しない（404 DISH_NOT_FOUND）
- 他ユーザーの料理へのアクセス（403 PERMISSION_DENIED）

**関連ストーリー**: US-03

### FR-04: 料理の更新（差分更新方式）

- 料理名・日付・カテゴリを後から修正できる
- 写真は個別に追加・削除でき、変更しない写真はそのまま維持される
- 更新後も写真の上限（最大3枚）は維持される
- 技術詳細: [design.md - 差分更新方式](design.md#画像更新ルール差分更新方式)

**正常系**: 200 OK — 更新後の料理情報が返却される
**エラーケース**:
- 画像枚数が上限を超過（400 IMAGE_LIMIT_EXCEEDED）
- 指定された画像が存在しない（404 IMAGE_NOT_FOUND）
- 画像が該当料理に属していない（403 IMAGE_NOT_OWNED）
- 他ユーザーの料理へのアクセス（403 PERMISSION_DENIED）

**関連ストーリー**: US-04

### FR-05: 料理の削除

- 不要になった料理記録を削除できる
- 削除した記録は一覧から非表示になる
- 削除ポリシーの詳細: [db-design.md](db-design.md)

**正常系**: 200 OK — 料理が論理削除される
**エラーケース**:
- 料理が存在しない（404 DISH_NOT_FOUND）
- 他ユーザーの料理へのアクセス（403 PERMISSION_DENIED）

**関連ストーリー**: US-05

### FR-06: 絞り込み検索

- カテゴリ・日付範囲で料理一覧を絞り込める
- 絞り込み条件はそれぞれ任意（組み合わせ可能）
- リクエスト/レスポンスの詳細: [design.md](design.md)

**正常系**: 200 OK — 条件に合致する料理一覧がフィルタされて返却される
**エラーケース**:
- カーソル値が不正（400 INVALID_CURSOR）

**関連ストーリー**: US-06

## 5. 非機能要件

### アクセス制御

- ユーザーは自分が登録した料理のみ操作できる
- 他ユーザーの料理への読み書きはすべて拒否される

### パフォーマンス

- 記録件数が増えても一覧取得の応答時間が大きく変わらない
- 写真アップロードはアプリ操作をブロックしない形で実現する

### データ整合性

- 料理データと画像ファイルの整合性が保たれる（DBに記録された画像は必ず参照可能）
- 一時的な不整合は許容するが、定期的に自動回収される

### 拡張性

- 将来的にレシピ紐付け・メモ追加などの機能拡張を妨げないデータ構造

## 6. スコープ外

| やらないこと | 理由 |
|---|---|
| メモ・評価・タグ付け機能 | MVP では記録と振り返りに集中し、付加的な情報管理は後続フェーズで検討 |
| レシピへの紐付け | レシピ機能自体が未実装のため、現時点では対象外 |
| 他ユーザーへの公開・シェア機能 | 個人利用を前提としており、ソーシャル機能は需要を確認してから検討 |
| カテゴリのCRUD | カテゴリはマスターデータとして管理し、ユーザーによる追加・編集は不要 |
| 削除した料理の完全消去（物理削除） | 論理削除で運用し、データ復元の余地を残す |
| 料理の統計・分析機能 | 記録の蓄積を優先し、分析機能は十分なデータが集まってから検討 |

## 7. 用語集

| 用語 | 説明 |
|------|------|
| 料理 | ユーザーが実際に作った食事の記録。名前・日付・カテゴリ・写真で構成される |
| 論理削除 | データを物理的に削除せず「削除済み」フラグを立てる方式。復元可能性を残す |
| カーソルベースページネーション | 「次のページ」を取得するためのカーソル値を使ったページング方式。大量データでも安定したパフォーマンスを実現する |
| 差分更新方式 | 更新時に変更する部分だけを指定する方式。画像は「追加する画像」と「削除する画像」を個別に指定する |
| Pre-signed URL | 一時的にS3オブジェクトへのアクセスを許可する署名付きURL。クライアントが直接S3に画像をアップロードする際に使用する |
